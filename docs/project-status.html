1. 프로젝트 개요

한 줄 소개

학생 PC에서 수업 중 방해되는 콘텐츠(소셜미디어·게임·동영상 등)를 자동 탐지하고 차단하는 시스템. 4단계 파이프라인(타이틀 → URL OCR → 본문 OCR → LLM)으로 오탐을 최소화하면서 실시간 차단 + 관리자 모니터링 기능을 제공한다.

핵심 가치

  • 로컬 LLM(Ollama) 사용 → 외부 API 없이도 동작, 프라이버시 보호
  • 4단계 캐스케이드 탐지 → 1ms부터 10초까지 비용에 맞게 최적화
  • 웹 기반 해제 인증 → 정당한 사유로 차단 해제 가능
  • Phase 2에서 중앙 관리 대시보드 + 클래스 단위 정책 관리로 확장

전반적 평가: Phase 1 로컬 시스템은 운영 가능한 수준. Phase 2는 S3/Lambda/API Gateway가 이틀 전 배포 완료되어 데이터 인제스트·관리 경로가 골격을 갖춤 — 다만 본래 DynamoDB로 가려던 저장소가 외부 의존 작업 미해소로 S3 JSON fallback 상태고, API 조회 라우트 추가도 같은 사유로 보류 중. 외부 의존이 풀리는 동안 작업 가능한 영역(EC2 LLM 서버, 학생 PC 클라이언트 어댑터, 스키마 설계, 로컬 통합 테스트)을 우선 진행.

탐지 파이프라인 (구현 완료)

1단계: 창 타이틀 블랙리스트 매칭   (즉시, ~1ms)        ─ TITLE_MATCH
2단계: URL 바 OCR + 키워드 매칭     (~100~300ms)       ─ URL_MATCH
3단계: 본문 OCR + 키워드 조합       (~200~400ms)       ─ KEYWORD_MATCH
4단계: Ollama 로컬 LLM 최종 판단   (~2~10초, 사양에 따라) ─ LLM 검증
       → BLOCK / ALLOW / UNSURE

주요 설계 결정 (Phase 2 대비 포함)

  • EventSink 추상화 — LocalJSONLSink만 사용 중이지만, RemoteSink/S3Sink 확장 가능한 구조로 작성
  • USE_CLOUD_LLM 플래그 — config.py 한 줄 변경으로 로컬↔클라우드 LLM 전환 가능
  • IP/MAC 자동 수집 — Phase 2 중앙 서버에서 학생 PC 식별용
  • LLM 호출 직렬화 — _llm_lock으로 동시 호출 방지 (메모리 폭증 방지)
  • UNSURE 처리 — 오탐보다 미탐 선택 (수업 흐름 방해 최소화)

데이터 자산

5. Phase 2 — 클라우드 마이그레이션 진행 중

설계·의사결정 단계

  • 전체 아키텍처 결정: EC2(LLM) + Lambda + S3 + DynamoDB + API Gateway
  • 데이터 저장소 분리: 이벤트 로그 → S3 / blocklist·allowlist → DynamoDB
  • 데이터 모델 스키마 확정 (DynamoDB blocklist/allowlist 키 구조) — PK=class_id, SK=list_type#value

AWS 인프라 구축 단계

  • S3 버킷 2개 생성 — focusguard-data(이벤트 로그), focusguard-admin(관리 페이지 정적 호스팅)
  • Lambda 함수 5개 배포 (Python 3.14, Zip, 2026-05-15경)
  • API Gateway focusguard-api-v2 배포 + 핵심 라우트 3개 구성
  • API Gateway 조회 라우트 추가 (fg-get-lists, fg-get-events) — 외부 의존 작업 대기 중
  • DynamoDB 테이블 생성 — fg-blocklist, fg-allowlist
  • EC2(t3.xlarge) 인스턴스 프로비저닝 + Ollama 설치

6. 다음 단계 (남은 작업)

우선순위 1 — 즉시 처리 (블로커 해제)

  • S3 JSON → DynamoDB 마이그레이션 계획 수립: 현재 fg-update-lists / fg-get-lists가 focusguard-data S3에 JSON으로 운영 중 — 동시 쓰기 race condition 위험 해소를 위해 DynamoDB로 전환 필요
  • API Gateway에 누락된 조회 라우트 추가 (fg-get-lists, fg-get-events)
  • blocklist / allowlist 스키마 확정 — 분리 테이블, PK=class_id, SK=list_type#value, GSI 미사용
  • DynamoDB 테이블 생성 — fg-blocklist, fg-allowlist

우선순위 2 — 클라이언트 통합

  • 학생 PC 클라이언트 → focusguard-api-v2 호출 어댑터 (event_logger.py S3Sink/RemoteSink)
  • 관리자 페이지(focusguard-admin) → 조회 API 연동 (목록 조회·이벤트 조회·해제 처리)
  • end-to-end 테스트: 학생 PC → API Gateway → Lambda → S3 적재 확인

우선순위 3 — EC2 LLM 서버 (병행 가능)

  • EC2 t3.xlarge 인스턴스 프로비저닝
  • Ollama 설치 + 모델 다운로드 (gemma3:4b 또는 경량 옵션)
  • CloudLLMClient.analyze() 구현 (EC2 LLM 서버 HTTP 호출)
  • USE_CLOUD_LLM=True 통합 테스트

우선순위 4 — DynamoDB 본격 통합

  • DynamoDB Local에서 검증한 코드를 실제 DynamoDB로 전환 (엔드포인트 교체)
  • fg-update-lists / fg-get-lists를 DynamoDB 기반으로 재구성
  • 관리자 대시보드 → DynamoDB 동기화 (blocklist 즉시 갱신)

우선순위 5 — 운영 안정화

  • API Gateway CloudWatch Logs 활성화
  • S3 로그 Athena 쿼리 템플릿 작성
  • EC2 LLM 서버 모니터링 (CloudWatch Agent or 외부 헬스체크)
  • S3 라이프사이클 정책 (오래된 로그 자동 삭제)

리스크

  • 외부 의존 작업 대기 → DynamoDB 이전·API 라우트 추가가 외부 처리 일정에 묶여 일정 압박 가능 → 그 사이 S3 JSON fallback으로 운영은 계속, 작업 가능한 영역(스키마 설계·로컬 통합 테스트·클라이언트 어댑터)을 우선 진행

  • 학생 PC 인터넷 차단 시 → API Gateway 호출 실패 → 로컬 fallback 모드 필요 (TODO)
  • MAC 주소 PII 처리 → 외부 전송 금지, AWS 내부 저장에 대해서도 가이드 정립 필요
